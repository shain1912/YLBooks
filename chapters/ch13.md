# 13장 캘리브레이션과 지연 보정

12장에서 판정 시스템을 완성하고 처음으로 실제 플레이를 해 봤을 때, 분명히 박자에 맞게 눌렀는데도 GOOD이나 MISS가 뜨는 상황을 겪으셨을 겁니다. 손이 늦은 것도 아니고, 리듬 감각이 없는 것도 아닌데 결과가 이상합니다. 범인은 여러분의 박자 감각이 아니라 **지연(latency)**입니다. 이번 장에서는 지연이 어디서 발생하는지 이해하고, 자동으로 측정해서 게임에 반영하는 캘리브레이션 시스템을 만듭니다. 이 장을 마치면 PERFECT 비율이 30%대에서 70%대로 훌쩍 뛰어오릅니다.

---

## 13.1 박자 맞췄는데 왜 GOOD이 뜰까 — 지연이라는 보이지 않는 적

리듬게임을 처음 만들면 누구나 겪는 당혹감이 있습니다. 눈으로 보기에 완벽하게 판정선에 노트가 닿았을 때 키를 눌렀는데, 결과 화면에는 GOOD 아니면 MISS가 잔뜩 쌓여 있습니다. 여러 번 해도 마찬가지입니다. "내가 이렇게 박자 감각이 없었나?" 하는 자책이 밀려옵니다.

그런데 사실 여러분은 박자를 맞추고 있었습니다. 문제는 여러분이 **들은 소리**와 게임이 **기록한 소리 발생 시각** 사이에 시간 차이가 있다는 것입니다. 노래는 컴퓨터 내부에서 이미 재생되고 있지만, 그 소리가 스피커나 헤드폰을 통해 여러분의 귀에 도달하기까지 짧지 않은 시간이 걸립니다. 동시에, 여러분이 키링이나 키보드를 눌렀을 때 그 신호가 컴퓨터에 도달하는 데도 시간이 걸립니다.

> **그림 13.1 — "보정 전 vs 후 — PERFECT 비율 30% → 75%"**
> *도식: 막대그래프 2개 좌우 배치. 왼쪽 '보정 전' — PERFECT 30%, GOOD 48%, MISS 22%. 오른쪽 '보정 후' — PERFECT 75%, GOOD 20%, MISS 5%. 화살표로 변화 강조.*

이 두 지연이 합쳐지면 아무 보정도 없는 상태에서 실제로 여러분이 경험하는 오차는 50~150ms에 달합니다. 12장에서 만든 판정 창이 PERFECT ±45ms, GOOD ±90ms였던 것을 기억하시나요? 100ms의 지연이 있다면 아무리 정확히 누르더라도 항상 GOOD 구간 끄트머리나 MISS를 맞힐 수밖에 없습니다.

이 문제의 해법은 간단합니다. 지연이 얼마인지 측정해서, 그만큼 미리 당겨서 판정하면 됩니다. 마치 멀리 있는 사람에게 공을 던질 때 현재 위치가 아니라 도착할 위치를 겨냥하는 것처럼요. 그 측정과 반영을 자동으로 해 주는 것이 캘리브레이션입니다.

## 13.2 두 가지 지연 분리하기 — 오디오 vs 입력

지연에는 크게 두 가지 경로가 있습니다. **오디오 출력 지연**과 **입력 지연**입니다. 각각 어디서 얼마나 생기는지 알면 우리가 무엇을 측정해야 하는지가 명확해집니다.

> **그림 13.2 — "지연 두 가지 — 오디오 출력 + 입력"**
> *도식: 위쪽 경로 — 컴퓨터(노래 재생) → 스피커/헤드폰 → 귀 (오디오 X ms). 아래쪽 경로 — 키링/키보드 → USB/블루투스 → 컴퓨터(pygame 이벤트) (입력 Y ms). 두 경로가 합쳐지는 화살표.*

**오디오 출력 지연**은 운영체제의 오디오 버퍼, 드라이버, 출력 장치를 거치며 발생합니다. 유선 3.5mm 이어폰이나 헤드폰은 대개 5~20ms 수준으로 무시할 만합니다. USB 오디오 인터페이스는 드라이버 설정에 따라 5~40ms, 그리고 블루투스 헤드폰은 100~200ms가 발생합니다.

> **그림 13.3 — "헤드폰 종류별 지연 비교 표"**
> *표: 유선 3.5mm / 평균 지연 5~20ms / 추천(★★★). USB 오디오 / 평균 지연 10~40ms / 추천(★★☆). 블루투스 / 평균 지연 100~200ms / 주의(★☆☆) — "리듬게임에는 권장하지 않음" 메모.*

블루투스 헤드폰의 100~200ms는 리듬게임에 치명적입니다. 그 정도면 음악이 귀에 들리는 시각이 게임의 내부 기준보다 거의 한 박자 뒤처집니다. 보정으로 어느 정도 커버는 가능하지만, 근본적으로 유선 연결을 권장합니다.

**입력 지연**은 키링이나 키보드의 신호가 USB 폴링을 거쳐 운영체제에 도달하고, pygame 이벤트 큐에 들어오기까지의 시간입니다. 일반 USB 키보드는 8~16ms(폴링 주기 기준), 우리가 쓰는 키링 HID 장치도 비슷한 범위입니다.

좋은 소식은, 이 두 지연을 따로 측정할 필요가 없다는 점입니다. 학생이 소리를 듣고 키를 누르는 전체 경험을 측정하면, 두 지연이 합산된 결과가 자연스럽게 나옵니다. 그 합산 값 하나로 모든 보정이 가능합니다.

## 13.3 캘리브레이션 화면 만들기 — 메트로놈 두드리기

이제 실제 측정 화면을 만들어 봅니다. 기본 아이디어는 간단합니다. 음악 선생님이 손뼉을 "하나, 둘, 셋, 넷" 치면 학생이 그 박자에 맞춰 따라치는 것처럼, 게임이 정확히 1초마다 "딱" 소리를 내고 학생이 그 소리에 맞춰 D키를 누릅니다. 학생이 키를 누른 시각과 소리가 재생된 시각의 차이가 바로 우리가 찾는 지연값입니다.

> **그림 13.4 — "캘리브레이션 화면 와이어프레임"**
> *도식: 게임 창 안에 중앙 상단 안내문 "메트로놈 박자에 맞춰 D키를 8번 누르세요". 가운데 큰 원(박자에 맞춰 깜빡임). 하단 진행 바 "3 / 8". 배경 어두운 색, 원 흰색/회색 교차.*

Claude Code에 이 화면을 요청할 때 측정 알고리즘을 함께 명시하면 더 정확하게 원하는 코드를 얻을 수 있습니다.

> **그림 13.5 — "Claude Code에 캘리브레이션 화면 부탁"**
> *스크린샷: Claude Code 프롬프트 창. 입력 내용 — "1초마다 beep 소리를 내는 캘리브레이션 화면을 만들어 주세요. beep 재생 시각을 beat_time으로 기록하고, 플레이어가 D키를 누르면 press_time을 기록합니다. diff = press_time - beat_time 을 리스트에 누적해서 8회 완료 시 평균을 반환합니다."*

Claude Code가 만들어 준 캘리브레이션 화면의 핵심 코드는 아래와 같습니다.

```python
import pygame
import json
import time

def run_calibration(screen, clock, beep_sound):
    """
    1초 간격 메트로놈에 맞춰 D키를 8번 누르게 하고
    평균 오프셋(ms)을 반환한다.
    양수 = 플레이어가 늦게 누름 / 음수 = 일찍 누름
    """
    diffs = []
    beat_interval = 1.0          # 1초마다 비프
    next_beat_time = time.time() + 1.0
    last_beat_time = None
    total_beats = 8
    font = pygame.font.SysFont(None, 48)
    small_font = pygame.font.SysFont(None, 32)
    flash = False                 # 원 깜빡임 상태

    while len(diffs) < total_beats:
        now = time.time()

        # 비트 발생
        if now >= next_beat_time:
            beep_sound.play()
            last_beat_time = next_beat_time
            next_beat_time += beat_interval
            flash = True

        # 화면 그리기
        screen.fill((20, 20, 30))
        msg = font.render(
            f"메트로놈에 맞춰 D키를 누르세요 ({len(diffs)}/{total_beats})",
            True, (220, 220, 220)
        )
        screen.blit(msg, msg.get_rect(center=(screen.get_width()//2, 80)))

        circle_color = (255, 255, 255) if flash else (80, 80, 100)
        pygame.draw.circle(screen, circle_color,
                           (screen.get_width()//2, screen.get_height()//2), 60)

        # 진행 바
        bar_w = int(screen.get_width() * 0.6 * len(diffs) / total_beats)
        pygame.draw.rect(screen, (80, 180, 120),
                         (screen.get_width()//5, screen.get_height()-80,
                          bar_w, 20))

        pygame.display.flip()
        flash = False  # 다음 프레임에서 꺼짐

        # 이벤트 처리
        for event in pygame.event.get():
            if event.type == pygame.QUIT:
                return None
            if event.type == pygame.KEYDOWN and event.key == pygame.K_d:
                if last_beat_time is not None:
                    press_time = time.time()
                    diff_ms = (press_time - last_beat_time) * 1000
                    # 너무 동떨어진 입력은 무시(±600ms 초과)
                    if abs(diff_ms) < 600:
                        diffs.append(diff_ms)

        clock.tick(120)

    return sum(diffs) / len(diffs)  # 평균 오프셋 반환
```

> **그림 13.6 — "메트로놈이 째깍거리는 캘리브레이션 화면 (실제 동작)"**
> *스크린샷: pygame 창. 중앙에 흰색 큰 원이 박자에 맞춰 밝게 빛나는 한 컷. 상단 안내 텍스트 "메트로놈에 맞춰 D키를 누르세요 (3/8)". 하단 진행 바 37.5% 채워진 상태.*

이 함수를 실행하면 1초마다 소리가 나고, 학생이 D키를 누를 때마다 오차를 측정합니다. 8번 모두 완료되면 평균 오프셋을 밀리초 단위로 반환합니다. 비트 직후 ±600ms 이내 입력만 받아들이기 때문에, 잘못 누른 입력이나 완전히 박자를 놓친 타격은 자동으로 걸러집니다.

## 13.4 평균 오프셋 계산과 저장

캘리브레이션 함수가 반환한 평균 오프셋을 어딘가에 저장해 두어야 게임을 다시 시작해도 값이 유지됩니다. 가장 단순하면서도 편리한 방법이 `settings.json` 파일입니다. 이미 게임 창 크기나 볼륨 설정을 담아 두기에 적합한 형식이고, 사람이 직접 열어서 값을 수정하기도 쉽습니다.

> **그림 13.7 — "8번 두드린 결과 — 평균 +73ms"**
> *스크린샷: pygame 창. 캘리브레이션 완료 후 결과 화면. "측정 완료!" 큰 텍스트. "평균 오프셋: +73ms" 강조 표시. "이 값이 settings.json에 저장됩니다." 안내문.*

오프셋 저장과 불러오기 코드는 다음과 같습니다.

```python
import json
import os

SETTINGS_PATH = "settings.json"

def load_settings():
    """settings.json을 읽어 dict로 반환. 없으면 기본값 반환."""
    if os.path.exists(SETTINGS_PATH):
        with open(SETTINGS_PATH, "r", encoding="utf-8") as f:
            return json.load(f)
    return {"user_offset_ms": 0, "volume": 0.8}

def save_offset(offset_ms: float):
    """user_offset_ms 키를 settings.json에 저장."""
    settings = load_settings()
    settings["user_offset_ms"] = round(offset_ms, 1)
    with open(SETTINGS_PATH, "w", encoding="utf-8") as f:
        json.dump(settings, f, ensure_ascii=False, indent=2)
    print(f"[캘리브레이션] 오프셋 저장: {offset_ms:.1f}ms")
```

저장 후 `settings.json` 파일을 열면 아래처럼 보입니다.

```json
{
  "volume": 0.8,
  "window_width": 480,
  "window_height": 800,
  "user_offset_ms": 73.0
}
```

> **그림 13.8 — "settings.json에 저장된 보정값"**
> *스크린샷: 코드 에디터(VS Code). settings.json 파일. `"user_offset_ms": 73.0` 줄 노란색 배경 강조. 오른쪽 여백에 "여기가 핵심!" 주석 말풍선.*

게임 메인 진입 시 `load_settings()`를 호출해 `user_offset_ms` 키가 0인지 확인합니다. 값이 0이라면, 즉 아직 캘리브레이션을 한 번도 하지 않은 상태라면 자동으로 캘리브레이션 화면을 먼저 띄웁니다. 한 번 측정해 저장하면 다음 실행부터는 그냥 게임 화면으로 바로 들어갑니다.

```python
def main():
    settings = load_settings()

    # 첫 실행이면 자동 캘리브레이션
    if settings.get("user_offset_ms", 0) == 0:
        offset = run_calibration(screen, clock, beep_sound)
        if offset is not None:
            save_offset(offset)
            settings["user_offset_ms"] = offset

    user_offset_ms = settings["user_offset_ms"]
    # ... 게임 루프 시작
```

이렇게 하면 새 컴퓨터에서 처음 실행하거나, 헤드폰을 바꿔서 지연이 달라졌을 때 메뉴에서 캘리브레이션을 다시 돌리기만 하면 됩니다.

## 13.5 게임에 보정값 반영

이제 측정한 오프셋을 실제 판정에 적용할 차례입니다. 핵심은 딱 한 줄입니다. 12장에서 만든 `judge()` 함수를 기억하시나요? 플레이어가 키를 누른 시각(`press_time`)과 노트의 이상적인 타이밍(`note_time`) 사이의 차이를 계산해서 PERFECT·GOOD·MISS를 결정했습니다.

보정은 간단합니다. `press_time`을 그대로 쓰는 대신, `user_offset_ms`만큼 당긴 값을 씁니다. 지연 때문에 늦게 눌렸다면 그만큼 앞으로 당겨서 계산하면 판정이 올바르게 됩니다.

> **그림 13.9 — "judge() 함수에 보정값 반영 — 한 줄 추가"**
> *스크린샷: 코드 에디터. judge() 함수 본문. `press_time -= user_offset_ms / 1000` 줄 노란 배경 강조. 오른쪽 여백에 손으로 쓴 듯한 "딱 한 줄!" 메모.*

수정 전후 코드를 비교하면 다음과 같습니다.

```python
# --- 12장 원본 ---
def judge(press_time: float, note_time: float) -> str:
    diff = abs(press_time - note_time) * 1000  # ms 단위
    if diff <= 45:
        return "PERFECT"
    elif diff <= 90:
        return "GOOD"
    else:
        return "MISS"


# --- 13장 보정 추가 ---
def judge(press_time: float, note_time: float,
          user_offset_ms: float = 0.0) -> str:
    press_time -= user_offset_ms / 1000   # ← 이 한 줄이 전부
    diff = abs(press_time - note_time) * 1000
    if diff <= 45:
        return "PERFECT"
    elif diff <= 90:
        return "GOOD"
    else:
        return "MISS"
```

함수 인자에 `user_offset_ms`를 추가하고, 첫 줄에 빼 주는 것이 전부입니다. 기본값을 `0.0`으로 지정했기 때문에 캘리브레이션을 하지 않은 상태에서도 동작이 달라지지 않습니다. 기존 코드를 거의 건드리지 않고 새 기능을 얹었다는 점에서 설계가 우아합니다.

이 함수를 호출하는 쪽에서는 게임 루프가 시작되기 전에 `load_settings()`로 불러온 `user_offset_ms`를 변수에 담아 두고, 매 판정마다 넘겨주면 됩니다 (그림 13.9 참조).

```python
# 게임 루프 내부, 키 입력 처리 부분
for event in pygame.event.get():
    if event.type == pygame.KEYDOWN:
        lane = key_to_lane(event.key)
        if lane is not None:
            press_time = time.time()
            nearest_note = get_nearest_note(lane)
            if nearest_note:
                result = judge(press_time,
                               nearest_note.time,
                               user_offset_ms)   # ← 한 줄 추가
                update_score(result)
                nearest_note.hit = True
```

기존 코드에서 달라진 부분은 `judge()` 호출에 `user_offset_ms` 인자를 추가한 것뿐입니다. 12장에서 공들여 짠 노트 낙하 로직, 판정 창 크기, 점수 계산 로직은 전혀 손대지 않았습니다.

## 13.6 결과 확인과 다음 장

이제 캘리브레이션을 마친 뒤 12장 차트를 다시 플레이해 봅니다. 보정 전에는 PERFECT 30%, GOOD 48%, MISS 22% 정도였다면, 보정 후에는 완전히 다른 결과가 기다리고 있습니다.

> **그림 13.10 — "보정 후 같은 차트 다시 플레이 — PERFECT 비율 상승"**
> *스크린샷: 게임 결과 화면. PERFECT 78%, GOOD 18%, MISS 4% 표시. 상단에 "CALIBRATED" 배지. 전체적으로 밝고 성취감 있는 배색.*

PERFECT 78%는 처음 봤을 때와 비교해 극적인 차이입니다. 변경한 코드는 단 두 군데, `judge()` 함수 인자 한 줄과 호출부 한 줄이 전부였습니다. 지연을 이해하고 측정해서 올바르게 반영했기 때문에 이런 결과가 나옵니다.

한 가지 더 생각해 볼 상황이 있습니다. 친구가 놀러 왔을 때 같은 컴퓨터로 플레이하면 어떻게 될까요?

> **그림 13.11 — "친구가 와서 같은 컴퓨터로 플레이 — 다시 캘리브레이션"**
> *도식: 왼쪽 학생 A — user_offset_ms: 73ms (유선 이어폰 사용). 오른쪽 학생 B — user_offset_ms: 112ms (블루투스 헤드폰 사용). 가운데 같은 컴퓨터 아이콘. 각자 다른 숫자가 settings.json에 저장되는 화살표.*

친구는 블루투스 헤드폰을 쓰기 때문에 지연값이 다릅니다. 메뉴에서 '캘리브레이션 다시 하기'를 선택해서 측정하면, `settings.json`의 `user_offset_ms`가 친구의 환경에 맞게 덮어쓰입니다. 그 뒤 플레이하면 친구도 최적의 판정을 경험합니다. 이처럼 캘리브레이션은 장치가 바뀌거나 사용자가 바뀔 때마다 한 번씩 돌려 주면 됩니다.

> **그림 13.12 — "13장 완성 — 박자가 정확히 맞는 플레이"**
> *촬영 지시: 키링을 쥔 손이 화면 아래에, 모니터에 PERFECT 콤보 폭발 이펙트가 가득 찬 장면. 표정은 안 나와도 좋고, 화면 글로우와 손의 역동적인 구도를 살릴 것.*

13장에서 배운 내용을 정리하면 다음과 같습니다. 리듬게임의 판정 오차는 오디오 출력 지연과 입력 지연이 합쳐져 발생합니다. 캘리브레이션 화면에서 메트로놈 소리에 맞춰 D키를 8번 두드리면 평균 오프셋을 자동으로 계산할 수 있습니다. 이 값을 `settings.json`에 저장해 두면 다음 실행 시 자동으로 반영되고, `judge()` 함수에서 `press_time -= user_offset_ms / 1000` 한 줄로 모든 보정이 완료됩니다.

다음 14장에서는 드디어 여러분 자신의 음악으로 차트를 만드는 방법을 배웁니다. 내가 좋아하는 노래의 BPM을 분석하고, 노트 배치를 직접 설계해서 세상에 단 하나뿐인 커스텀 차트를 완성합니다. 13장에서 판정 정확도를 잡아 놨으니, 이제 어떤 차트를 올려도 제대로 된 플레이를 즐길 수 있습니다.
